## 바이브 코딩에서 편했던 점

### 1. 즉시 검증 사이클

```
문제 발견 → 코드 수정 → 서버 자동 리로드 → 확인
(평균 30초 ~ 3분)
```

**실제 사례:**
- Chart.js 캔버스 무한 확장 → div 고정 높이 추가 → **30초 해결**
- AI 할일 영어 출력 → 프롬프트에 한국어 강제 추가 → **3분 해결**
- 마일스톤 진행률 100% 이상 → DB 누락분 확인 + 추가 → **5분 해결**

워터폴이었다면: 설계서 수정 → 리뷰 → 승인 → 반영 → 테스트 (최소 며칠)

### 2. 실시간 핑퐁으로 방향 즉시 수정

| 제이님 피드백 | 반영 시간 | 결과 |
|-------------|----------|------|
| "서버 상시 가동 못 해" | 5분 | cc() 세션 연동 제거, SSH+Claude만 남김 |
| "AI 할일이 영어로 나와" | 3분 | GPT 프롬프트 한국어 강제 |
| "1인인데 알림 시스템 왜 추천해" | 5분 | 프롬프트에 1인/로컬 맥락 추가 |
| "마일스톤 100%가 이상해" | 1분 | 원인 설명 (DB 미동기화) |
| "기존 영어 할일도 바꿔줘" | 3분 | DB 직접 UPDATE 7건 |

**핵심**: 피드백 → 코드 반영 → 확인이 **같은 대화 안에서** 끝남.

### 3. 앞 스프린트가 뒤 스프린트의 기반

```
S0: ob_projects (project_id 탄생)
  ↓
S1~S9: 모든 테이블이 project_id FK로 자연스럽게 연결
  ↓
대시보드: GROUP BY project_id 한 번으로 전체 통계 집계
```

테이블 설계를 한번에 안 하고 스프린트마다 필요한 만큼만 추가했는데, `project_id` 중심 구조가 일관되게 유지됨. **규칙 하나(ob_ 접두사 + project_id FK)만 지켰더니 13개 테이블이 깔끔하게 연결됨.**

### 4. CONTEXT.md가 살아있는 지식 베이스

스프린트마다 실패 패턴과 기술 결정을 바로 기록 → 다음 스프린트에서 같은 실수 방지.

**기록이 살린 사례:**
- alembic `sslmode` → `ssl` 치환 필요 (한번 삽질 → 기록 → 다시 안 겪음)
- seed.py 볼륨 마운트 누락 (Docker COPY vs 마운트 차이 기록)
- Git Bash psql SSL 인증 실패 (psql 문제지 앱은 정상 — 기록으로 혼란 방지)

### 5. Docker + RDS로 환경 분리 걱정 없음

```
로컬 코드 수정 → 볼륨 마운트로 자동 반영 → uvicorn --reload
DB는 RDS라 컨테이너 재시작해도 데이터 유지
```

"환경 맞추기"에 시간 안 쓰고 **기능 구현에만 집중** 가능했음.

## 불편했던 점 / 교훈

### 1. 빈 테이블 5개 — "미리 만든 것"의 대가

S7에서 인프라 관리를 한 스프린트에 묶으면서 `ob_deployments`, `ob_db_migrations`, `ob_server_snapshots`를 한번에 생성. 결과적으로 아직 데이터 0건.

**교훈**: 순수 애자일이면 "쓸 때 만든다." S7을 세분화했으면 불필요한 테이블이 없었을 것.

### 2. 마일스톤 DB 동기화 밀림

코드는 S9까지 갔는데 DB 마일스톤은 S3에서 멈춰있었음 → AI 할일이 "타임라인 구현하세요"를 계속 추천.

**교훈**: 스프린트 완료 시 마일스톤 상태도 같이 업데이트하는 루틴 필요. (지금은 작업할 때마다 추가하기로 함)

### 3. seed.py와 실제 DB의 괴리

seed.py는 초기 상태(S0~S3)로 고정 → 프로젝트가 S9까지 성장하면서 seed.py가 무의미해짐. 마일스톤, 에이전트를 직접 DB에 INSERT하는 일이 많았음.

**교훈**: seed.py는 "최초 1회용"으로만 사용하고, 이후 데이터 변경은 API 또는 마이그레이션으로.

### 4. 민감정보 관리 후순위

docker-compose.yml에 DB 비밀번호, GitHub 토큰이 하드코딩 → GitHub push 직전에 발견해서 .env 분리.

**교훈**: S0에서 .env 분리를 확정하고 시작했어야 함. 로컬이라 괜찮았지만, push 순간 위험.

## 애자일 vs 워터폴 — 이 프로젝트 기준 비교

| 상황 | 애자일 (실제) | 워터폴 (가정) |
|------|-------------|-------------|
| 첫 화면 볼 때까지 | S0 당일 | 전체 설계 후 최소 1주 |
| "영어 → 한국어" 변경 | 3분 | 요구사항 변경 → 재설계 → 재구현 |
| "cc() 세션 연동 제거" | 5분 | 설계된 시스템 롤백 (영향 분석 필요) |
| 13개 테이블 완성 | 9일 (점진적) | 설계만 며칠 + 구현 며칠 |
| 빈 테이블 문제 | 5개 (과잉 생성) | 0개 (전부 설계대로) |
| 방향 전환 비용 | 거의 0 | 설계 변경 = 대규모 수정 |

**결론**: 1인 프로젝트 + 바이브 코딩에서 애자일은 **압도적으로 유리**. 워터폴이 나은 유일한 포인트는 "빈 테이블 없음"인데, 그건 빈 테이블 5개의 비용(거의 0)으로 상쇄됨.

## 바이브 코딩 특유의 패턴

### 핑퐁 설계

```
Claude: "이렇게 설계할까요?"
제이님: "응" 또는 "아니 이렇게"
Claude: 바로 구현 + 확인
→ 설계와 구현이 분리되지 않음
```

### 실시간 디버깅

```
서버 에러 → 로그 확인 → 원인 파악 → 수정 → 확인
(전통적: 버그 리포트 → 재현 → 분석 → 수정 → 배포 → 확인)
```

### 컨텍스트 연속성

같은 대화 안에서 "cc() 단순화 → 트렌드 차트 → AI 프롬프트 → 스키마 보강 → 에러 페이지 → 스파크라인 → GitHub push"까지 한 흐름으로 진행. **컨텍스트 전환 비용 = 0**.
